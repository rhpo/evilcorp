# EvilHide: Steganography Implementation Report

## 1. Introduction
Steganography is the practice of concealing a message, file, or image within another file, message, or image. Unlike cryptography, which focuses on making a message unreadable to unauthorized parties, steganography focuses on hiding the very existence of the message. In the **EvilCorp** project, the **EvilHide** module provides two distinct steganographic methods:
- **Image-in-Image Steganography:** Hiding a secret image inside a cover image.
- **Text-in-Image Steganography:** Hiding a plain text message inside an image.

Both implementations use the **Least Significant Bit (LSB)** substitution technique, which is a common and effective method for digital steganography.

## 2. Image-in-Image Steganography Algorithm

### 2.1 The Concept of LSB Substitution
Digital images are composed of pixels, where each pixel is typically represented by three or four color channels (Red, Green, Blue, and sometimes Alpha). In a standard 24-bit image, each channel uses 8 bits, representing values from 0 to 255.

The LSB algorithm works by replacing the least significant (rightmost) bits of the cover image's pixel values with the most significant (leftmost) bits of the secret image's pixel values. Because the LSBs contribute very little to the overall color of a pixel, these changes are virtually imperceptible to the human eye.

### 2.2 Encoding Process (Fusion)
1. **Mask Preparation:** The user provides an 8-bit binary mask (e.g., `11111100`). The number of zeros in the mask determines how many bits per channel will be used for hiding (in this case, 2 bits).
2. **Bit Selection:**
   - For each pixel in the **Cover Image**, we clear the specified number of LSBs using the mask.
   - For each pixel in the **Secret Image**, we take the specified number of most significant bits (MSBs).
3. **Merging:** The MSBs of the secret pixel are shifted and combined with the masked cover pixel.
4. **Result:** The resulting "Composite Image" looks identical to the cover image but contains the secret image's high-order data in its LSBs.

### 2.3 Decoding Process (Extraction)
1. **Bit Isolation:** Using the same mask (or rather, the inverse of it), we extract only the LSBs from the composite image.
2. **Scaling:** The extracted bits are shifted back to the MSB positions to reconstruct the most significant part of the original secret image.
3. **Reconstruction:** The extracted pixels are used to generate the decoded image. Note that some quality is lost from the secret image (specifically, the detail that was in its own LSBs), but the core image remains recognizable.

## 3. Text-in-Image Steganography Algorithm

### 3.1 Data Preparation
Unlike images, text data is non-visual and must be serialized into a bitstream before it can be hidden.
1. **Encoding:** The text is converted into a byte array using UTF-8 encoding.
2. **Metadata:** To ensure successful extraction, the length of the text (in bytes) is prepended to the data as a 32-bit integer.
3. **Binary Stream:** The total data (4 bytes for length + $N$ bytes for text) is treated as a continuous sequence of bits.

### 3.2 Embedding Logic
1. **Capacity Check:** The algorithm ensures the cover image has enough pixels to store the total number of bits.
2. **LSB Substitution:** One bit of the stream is embedded into the LSB of one color channel (e.g., the Blue channel) of each pixel. Alternatively, it can rotate through Red, Green, and Blue channels for higher capacity or better secrecy.
3. **Traversal:** The algorithm iterates through pixels sequentially (left-to-right, top-to-bottom) until all bits are hidden.

### 3.3 Recovery Logic
1. **Length Retrieval:** The first 32 pixels are read to extract the LSBs of the chosen channel, reconstructing the 32-bit integer that represents the text length.
2. **Data Retrieval:** Based on the retrieved length, the algorithm continues reading the subsequent pixels' LSBs to collect the rest of the bitstream.
3. **Reconstruction:** The bitstream is converted back into bytes, and the UTF-8 string is reconstructed.

## 4. Performance and Security Considerations

### 4.1 Implementation Efficiency
In **EvilHide**, performance is optimized using the `.NET` `LockBits` method and `unsafe` pointers. This allows for direct memory manipulation of the image's pixel buffer, avoiding the slow `GetPixel` and `SetPixel` methods. This ensures that even large 4K images can be processed in milliseconds.

### 4.2 Robustness and Detection
While LSB steganography is visually stealthy, it is susceptible to **Steganalysis** (statistical analysis of bit distributions). It is also non-robust, meaning that applying compression (like JPEG), resizing, or cropping the image will likely destroy the hidden data. For maximum reliability, images should be handled in lossless formats like **PNG** or **BMP**.
